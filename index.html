Requirements

  • Tracking accuracy: 50m
  • Battery-lifetime: 10 years
  • Communication range: 10km
  • Simple visualization of power consumption
  • Optimization parameters
    1. Range of communication
    2. Power consumption
    3. Modularity
The requirements of the project are.

First, the accuracy of the tracker should be within 50m.
This was chosen as you would be able to get a general idea of where the device is.

It should be able to keep running for at least 10 years on a single CR2032 battery.
10 years was chosen, as this is the target of most LPWAN devices.

It should be able to connect to the network from at least 10km away.
This was chosen, as it would allow for better coverage.

We should be able to get a simple overview of the power consumption of the system.
This would aid us in reducing the power consumption.

We also want to optimize the project for different parameters, these are range, power consumption, and modularity.
This should be the order of importance when selecting components and writing the code.

The reason range is above power consumption is that we want good coverage, otherwise short range networks would be better.

Picking Components

  • Needed components:
    1. MCU
    2. Communication
    3. Tracker
  • STM32
  • LR1110
  • BC660K-GL
Three components are needed, an MCU, a communication device and a tracking device.

Based on trying to optimize for power consumption in both running and sleep mode, we chose to go with the STM32 line, specifically the STM32U0 and STM32U3.

Then, to communicate over LoRa, we chose the LR1110, which will also be used for tracking, as it contains a WiFi and GNSS receiver. Because of this, we need to always have the LR1110 on the project, to get it to work.

Finally, we have the BC660K-GL, which we use for NB communication.

The program consists of 5 main parts: Initialization, Join network, Scanning location, Send data, and exit.

---- NEXT ----
Each part of the code does a specific task. This task will change based on what we want to program to do. To do this we use defines to mask out specific parts of the code at compile time. This allows for more efficient execution compared to using normal if-statements, as masking out the code will not create any additional code.

Program Implementation

Initialization

  1. Reset LR1110
  2. Resume LR1110 communication
  3. Initialize BC660K-GL
  4. Connect to NB
  5. Initialize LR1110
Initialization handles setting up the LR1110 and BC660K-GL. Based on the settings, this includes resetting the LR1110, resuming its radio communication, connecting the BC660K-GL to NB, and initializing the LR1110.

Program Implementation

Join Network

  • Join LoRa
  • Connect MQTT
Joining the network is split into two paths: Joining LoRa or connecting to MQTT. The LoRa part does not guarantee that we are connected, this is checked later on. This allows us to initiate the scanning without having to wait for a response.

Program Implementation

Scan Location

  1. Scan WiFi
  2. Scan GNSS
  3. Wait for WiFi
  4. Wait for GNSS
  5. Is data valid?
    • No: Scan again and wait for result
    • Yes: Continue
Location scanning can be done with WiFi or GNSS. First we initiate the scanning, then we wait for a response. The GNSS scanning might return that there are no satellites, if this happens, we scan and wait again.

This is not split into two paths, as we allow both WiFi and GNSS scanning.

Program Implementation

Send Data - LoRa

  1. Wait for connection
  2. Try to send data
  3. If data transmission failed, rejoin network
We then send the scanned data. This is done over LoRa or NB.

For LoRa we first check if we have joined the network, if it failed, we try again until we succeed. We then try to send the data, first we check how many times we have failed at sending the data, if this is greater than our counter, we leave the network and rejoin. We then send the WiFi or GNSS data. If we failed, then we try again from the beginning, otherwise we continue.

This loop is created to harden the code, as we noticed when testing the system, it might sometimes be unable to send the data, at which point re-joining the network helped.

Program Implementation

Send Data - MQTT

  1. Send WiFi data
  2. Send GNSS data
Sending the data over NB is simply done by publishing the results to MQTT. All error handling is done in the send function, because of how we communicate with the BC660K-GL.

Program Implementation

Exit

  1. Leave LoRa
  2. Suspend LR1110 communication
The final part of the code exits the LoRa network and suspends the modem right before we sleep.

Program Implementation

Sleep

  • Different sleep modes used different places:
    1. Waiting for LR1110 communication
    2. Sleeping at the end after execution is done
There are two part of the code where we sleep, at the end and when waiting for the LR1110 to communicate.

---- NEXT ----
Waiting for the LR1110 should keep the program counter valid, which means we have to retain RAM and registers. Because of this, we can only enter XXX sleep.

---- NEXT ----
The final sleep should consume the least amount of power, therefor XXX sleep mode is used. This also has the effect of clearing RAM and resetting the program, thus we do not have to free any pointers or create an infinite while loop.

Profiling Code

One of the requirements was to easily profile code and find which parts of the code consume the most power. To do this, we created a program, which can take the measurements and code, and display the charge consumption per line.

This means we would first need to convert the code into specific sections. Then we need to convert the measurements into the consumption for each of these sections. We use two different scripts for this, one for converting code and the other for collecting measurements. Then a separate script was created, which could display these as the background color of the code.

The colors represent the fraction of how much a line takes to execute compared to the total charge consumed. This fraction is then gamma corrected, to make small differences in the not-so power consuming section be more visible.

The user can also highlight a line of code to get the charge consumed at that line at the bottom of the window.

Discussion

Power Consumption

  • Optimization flags
    • MCU waiting for LR1110 more
    • Favoring fast energy intensive instructions
  • Adding PU and leaving network
  • Not waiting for response
  • Optimal frequency
On the charge consumption, we can see that compiler flags seem to have no real effect, with the exception of using -O1. Since this is counterintuitive, more measurements would be needed before a definitive answer can be given. Some explanations would be that greater optimizations would make some parts of the code too fast, which means we would need to wait for the LR1110 more. The other explanation would be that optimization flags do not care about power, only execution time, since some flags might change the code to use more energy intensive instructions, this could increase the power consumed.

Adding a PU on NSS increases the average current consumption when sleeping, however this is due to the LR1110 now periodically sending and receiving messages while the MCU sleeps. We can combine this with leaving the network before sleeping, which removes this, and we can therefor see a great decrease in average charge consumed. This does not affect the charge consumed when running at all.

We also found that not waiting for a response from the LoRa network, would decrease the charge consumed, this is due to us no longer receiving the first acknowledge from the server.

Finally, we found out that the optimal frequency for the MCU CPU was around 524kHz. At this frequency, we execute the code fast enough for the LR1110 to not be wasting time unnecessarily, without wasting power more than necessary when sleeping.

Discussion

Battery-Lifetime

  • Charge decrease when reducing voltage on longer programs
  • When skipping listening, no charge reduction observed
    • Voltage reduction has no effect on WiFi scanning, which dominates when not waiting
    • Only one charge spike when transmitting data, which is the only reducable charge consumption
  • Skip listening did not work, no package received
  • Sleep current estimated, not measured
  • Sleep period very long, could be fixed with adding battery
Looking at the supply voltage graph, we see that reducing the voltage from 3.3V to 2.5V gives a noticeable decrease in charge consumed. This effect was not noticed as much when not waiting for RX response, which is probably due to it not being able to receive any packages.

We were also unable to measure the sleep current on the final PCB, as we did not have a PU on the NSS pin. This means the results are not as accurate as they do not account for all other components and swings in the current consumption.

From the estimated battery lifetime, we found that a sleep period of about 4 days would be needed, this is a little longer than what could be considered useable. We could add more batteries to reduce this period, with 4 batteries reducing it down to 1 day between executions. If a shorter period is needed, and we do not want to add additional batteries, then we would have to live with a shorter battery lifetime.

Conclusion

Achieved

  • Battery life of 10 years
  • Simple visualization of power consumption
  • Communication range of 10km
We were able to create a prototype of a tracker, which could last for 10 years of a single CR2032 battery, however only if we can accept a period of 4 days.

To help with optimizing the power consumption, we were able to create a program, which can visualize the charge consumed at each line. This program was successfully deployed and significant optimizations were made.

The tracker was able to communicate over protocols, which in theory should support up to 10km range. This was not experimentally concluded however.
